## Architecture — Locked In (do not re-litigate)

**Compile targets:**
- **Native HA Script (primary)** — every piston compiles to two files: an automation wrapper (triggers + conditions) and a script body (all action logic). Covers ~95% of real pistons. Zero external dependencies. Truly permanent after PistonCore uninstall.
- **PyScript (fallback)** — only for pistons that use `break`, `cancel_pending_tasks`, or `on_event`. Rare edge cases. Clearly labeled as a dependency.

**Why native scripts, not simple YAML:** Native HA scripts handle variables, all loop types, waits, if/then/else, and more. Simple YAML is a strict subset of what native scripts can do. There is no reason to maintain a separate YAML compiler.

**Global variables:** Stored as native HA helpers managed by the companion. Input_text for Text, input_number for Number, input_boolean for Yes/No, input_datetime for Date/Time. No globals.json file. Compiled pistons read helpers via standard HA template syntax.

**Minimum HA version:** 2023.1

**Other locked decisions:**
- Backend: Python FastAPI
- Frontend: Vanilla JS, HTML, CSS — no framework
- Piston storage: JSON files in Docker volume (`pistoncore-userdata/pistons/`)
- Compiler output: Jinja2 snippet templates assembled by Python compiler
- Default port: 7777
- HA integration: WebSocket API for capability data, companion for file writes
- Two Docker volume folders: `pistoncore-userdata/` and `pistoncore-customize/`
- `configuration.yaml` addition: `script pistoncore: !include_dir_merge_named scripts/pistoncore/`
- Theme: dark mode default, light mode option (user setting)

**UI — locked decisions:**
- Keywords: teal color
- Ghost text insertion points: always visible, dim/muted, inline in the document flow
- Right-click context menu: copy/cut/duplicate/delete/clear clipboard
- Test button: always "Live Fire ⚠" with confirmation dialog
- AND/OR: same indent level as conditions

**Compiler:**
- execute/end execute is a rendering artifact — not a JSON node
- Jinja2 templates are snippet-level, not whole-file. Python makes decisions, templates hold HA syntax.
- wait until time → `wait_for_trigger` with time trigger (not delay math, not wait_template polling)
- wait until time always emits a CompilerWarning (past-time hang behavior) — see COMPILER_SPEC Section 14
- wait until time shows a ⓘ tooltip in the editor — see FRONTEND_SPEC wait tooltip section
- Variable scope caveat: variables set inside loops and read outside emit a warning, not a rewrite
- Slug format: lowercase, underscores, max 50 chars, collision → append piston ID prefix
- Automation `id:` field = piston ID (never changes). Filename and `alias:` = slug (changes on rename).
- Renaming a piston changes the slug, filename, and alias — HA treats it as a new automation entity
- `continue_on_error: true` on all service calls in with_block (matches WebCoRE fire-and-forget behavior)
- `wait_for_state` uses `continue_on_timeout: true`; branch on timeout via `{{ not wait.completed }}` if block

**Sharing:**
- Snapshot (green) = anonymized, safe to share
- Backup (red) = full with entity mappings, personal restore only
- No central server, no short codes

**Validation pipeline:**
- Stage 1: internal checks in Docker (always runs on save)
- Stage 2: compile to sandbox
- Stage 3: yamllint on sandbox files
- Stage 4: REMOVED
- Stage 5: pass → deploy, fail → nothing written to production

**Run status:**
- PISTONCORE_RUN_COMPLETE event fires on script completion
- If not received within 5 minutes (configurable), status shown as "unknown" — never "Running" indefinitely

## ⚠ CRITICAL — Editor and Wizard: What Is Wrong and What Must Change

**The editor and wizard in the current `frontend/js/editor.js` and `frontend/js/wizard.js` are wrong and must be completely rewritten before any other frontend work.**

The user's single most important priority is the editor and wizard feeling right. Everything else comes after.

### What the current editor got wrong

The current editor looks like **Rule Machine** — boxed labeled sections (TRIGGERS box, CONDITIONS box, ACTIONS box). The user immediately and strongly rejected this. It is the wrong pattern entirely.

**The correct pattern is WebCoRE** — a single continuous code document, flowing top to bottom. No boxes. No panels. No labeled sections.

### What the editor must look like — from real WebCoRE screenshots

Before writing any editor code, Claude must ask the user to provide WebCoRE screenshots, OR reference the screenshots already provided in past sessions. The repo contains a `webcore_screenshots/` folder from Session 5 — read those first.

**The correct editor layout:**
- **Full-width code document** — the entire page is the script, no boxes
- **Left gutter** — line numbers, left edge
- **Right sidebar** — globals panel only, fixed width, always visible
- **Tiny toolbar** at the very top — small icon buttons only. Cancel / Delete / Save / Options at top right.
- **No separate Triggers / Conditions / Actions labeled panels** — these are not separate UI sections

**The document flows as one continuous script:**
```
/* comment header — piston name, author, dates, build number */

settings
end settings;

define
    device Light = Dining Light;
    + add a new variable
end define;

only when
    + add a new restriction

execute
    only when
        + add a new restriction
    if
        + add a new condition
    then
        + add a new statement
    else if
        + add a new condition
    else
        + add a new statement
    end if;
    + add a new statement
end execute;
```

**Rendering rules from screenshots:**
- Keywords (`settings`, `define`, `execute`, `if`, `then`, `else`, `end if;`, `only when`, `when true`, `when false`, `do`, `or`) — colored teal/green
- Device references in `{curly braces}` — colored distinctly (orange/teal)
- Attribute names — italic
- Ghost text (`+ add a new statement`, `+ add a new variable`, `+ add a new restriction`, `+ add a new condition`) — dim gray, small, always visible inline
- Currently selected line — full-width blue highlight across the document
- Line numbers — left gutter, muted
- The `only when` block appears BOTH at the top level (above `execute`) AND inside blocks
- `when true` / `when false` labels the branches inside if blocks IN THE EDITOR (not `then`)
- `then` is used in the status page read-only view and export format only

### What the wizard must look like — from real WebCoRE screenshots

**The correct wizard pattern:**

- Simple form layout — dropdown(s) + text field(s) + Add/Cancel buttons
- The editor document remains visible behind it (dimmed)
- Steps through the choices in a compact way — not card grids, not full-screen takeover
